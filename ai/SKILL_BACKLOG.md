- 2026-05-27 — Add a `writing-cross-functional-decision-docs` skill (type: new). Why: today we co-authored a phased rollout plan for the `dataApiRevokeOnCreateDefault` flag (Notion: "Revoke Default Grants Rollout Plan", id `36d5004b-775f-819d-b0f5-ecbd7d630ca2`; local draft: `~/supabase/docs/plans/2026-05-27-data-api-revoke-on-create-rollout-plan.md`). Started bloated (Notion AI: "reads like an analytics plan, not a rollout plan"), ended sharp after ~15 rounds. Generalizable principles to encode: (1) **Status callout at top** answers "where are we right now? what is this review asking for?" before anything else — burying current state in the Decision Log is wrong. (2) **Overview leads with customer impact**, not feature mechanic ("New projects no longer grant Data API privileges to anon/authenticated by default" beats "the flag flips the Grant default privileges checkbox"). (3) **Strawman thresholds + visual highlight + explicit "push back" framing** — vastly higher reviewer engagement than `TODO%` placeholders. Use `<span color="yellow_bg">` in Notion. Pair with a sentence like "these highlighted thresholds are a strawman — push back on the numbers themselves." (4) **Table altitude matches decision altitude.** 5-row phase table = strategy reviewers approve; 25-row step table = tracker for execution. Don't conflate; pick the right granularity for the doc's job, and offload tracking to a separate artifact (Decision Log, Linear, dashboard). (5) **Strict sequencing over hedge clauses** — no "Phase X complete OR trips diagnosed/remediated" escape hatches. Commit, amend later via Decision Log. Pre-written escape hatches signal you're already negotiating against your own plan. (6) **Cohort-anchored ramp descriptions** when progression is per-customer ("small SFP → larger SFP") not per-percentage — don't shoehorn into percentages just because tables look uniform. (7) **`<details>` toggles preserve analytical depth without bloating linear read** — great for diagnostic interpretation rules, known limitations, edge-case nuance, and anything that matters to one reviewer but distracts others. (8) **Footnotes with formula + worked example + justification** (not just definition). Justification preempts the "shouldn't this be different?" reviewer question. Example pattern: "Formula: `(control − test) / control`. Example: 2% relative regression on 80% baseline = test dropped to ~78.4%. Used for: rate-style metrics; scales fairly across cohorts." (9) **Pass-criterion framing**: relative regression for rate-style metrics (scales across cohorts with different baselines), absolute rise for near-zero metrics (avoids divide-by-near-zero distortion where 0.1→1.0 reads as 900% relative but is only 0.9pp absolute). (10) **Iterative deletion IS the work.** Final draft was ~50% shorter than first. The signal you're done editing is when each round produces deletions rather than additions. When a section's purpose has been absorbed by another section (e.g., metric definitions migrating from prose into a Guardrails table), delete the section cleanly rather than try to preserve every paragraph. (11) **Anti-pattern: bolded-title-then-explanation bullets** create a false sentence break and read unnaturally. Plain prose paragraphs or definition-style "**Field:** value" labels both work; mid-bullet sentence splits ("**Title.** explanation sentence") don't. (12) **Filter LLM doc feedback critically.** "Restructure into PRD template" is the default over-correction when only 1-2 surgical edits address the real issues. When LLM feedback says "restructure," ask which paragraphs specifically would move; if the answer is "all of it," the feedback is reaching for a template. Skill should encode these as a checklist + a worked before/after pair showing how a too-technical analytics-style draft becomes a stakeholder-ready rollout plan. Closely related to but distinct from `writing-pr-descriptions` (different artifact, similar voice discipline) and `writing-plans` (similar artifact, but typically internal-only and lower-stakes than cross-functional).

- 2026-07-14 — writing-skills / writing-great-skills: add two empirically-tested authoring rules discovered while TDD-ing the writing-as-sean skill (type: improve). (1) COUNTER-EXAMPLES TEACH THE BEHAVIOR THEY FORBID. A rule reading "write X, never Y" reliably produced Y. The em-dash rule failed 3 consecutive runs while the skill file contained a literal em-dash in its own "never write this" example; deleting the character from the file fixed compliance immediately, with no change to the rule's wording. Corollary: never put a banned token, phrase, or construction literally in a skill file, even inside a Don't column. Describe it or omit it. This is Pocock's negation principle, confirmed under test. (2) SEARCH RULES PASS, JUDGMENT RULES FAIL. "Don't narrate the investigation" failed 4 runs; the identical intent rewritten as "search your draft for 'I found, I traced, in one run, confirmed' and delete those whole sentences" passed first try. Any rule the agent can rationalize past ("this evidence IS load-bearing") should be restated as a literal string search with a zero count. (3) Placement matters more than wording: instructions in a numbered "Hard rules" block near the top were obeyed; the same instructions in a later "Self-check" section were skipped.
